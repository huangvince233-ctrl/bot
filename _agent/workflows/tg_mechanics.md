# 📱 Telegram 官方底层机制说明

> [!NOTE]
> 本文记录 Telegram 平台官方 API 的核心行为，帮助理解项目的备份报告、进度计数及发现流水线校验逻辑。

---

## 一、原始消息与媒体组 (Media Groups)

在 Telegram 协议层，"原始消息" (Raw Message) 是 API 传给程序的**最小物理单元**。

- **核心限制**：一个 `msg_id` 只能携带一个主媒体文件。
- **相册机制 (Albums)**：当用户一次性发送多张图/视频时，电报实际上生成了多条 `msg_id` 的独立消息，但通过 `grouped_id` (媒体组 ID) 将它们关联。
- **项目计数逻辑**：
  - **原始条数**：API 推出的物理消息总数（相册里的每一张都会被计入）。
  - **消息组数**：项目逻辑层按 `grouped_id` 合并后的逻辑帖。

---

## 二、消息 ID 步进与进度估算

Telegram 频道消息 ID 是**单调递增**的正整数。

### 1. ID 空洞 (Gaps)

如果管理员在频道中删帖，对应的 `msg_id` 会消失，形成空洞。

### 2. 进度分母生成策略

- **精确模式 (Exact)**：拉取最新 2000 条消息进行物理计数，剔除空洞，分母显示真实数字。
- **估算模式 (Estimate)**：当增量远超 2000 时，系统退化为采用 `(最新ID - 断点ID)` 的差值作为参考。此时进度条前缀会显示 `~` 符号，表示这仅是上限估量。

---

## 三、私有频道跳转链接协议

私有频道消息跳转不使用 `t.me/username/` 格式，而是采用：

```text
https://t.me/c/{频道ID去掉-100前缀}/{消息ID}
```

示例：

- 频道 ID：`-1001234567890`
- 消息 ID：`42`
- 链接：`https://t.me/c/1234567890/42`

---

## 四、流水线新鲜度感知机制 (V2.2 New)

Discovery Pipeline 获取“最新发现”的关键在于感知本地备份文件的变动。

### 1. 逻辑断点：Backup Label

系统读取数据库中记录的最新备份标签（如 `B3`、`B4`）。如果候选池元数据记录的仍是 `B3` 而数据库已到 `B4`，则视为过期。

### 2. 物理锚点：Max MTime

为了应对“同 ID 增量备份”（即 Backup Label 未变但内容已刷新）的情况，系统实时读取备份目录内 JSON 的 `Max MTime` (最后修改时间)。

- **逻辑**：即便标签相同，只要物理文件发生了 1 秒以上的写入偏移，系统也会侦测到“新鲜数据”，从而强制重跑 P0/P1，防止漏掉任何新挖掘到的词条。
