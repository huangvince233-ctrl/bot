# 🤖 Telegram Video Copilot 工作模式与工作流

> [!IMPORTANT]
>
> ## 🛡️ 开发者与 Agent 核心守则 (Developer & Agent Mandatory Rules)
>
> 为确保开发的稳定性和逻辑生效的即时性，在修改任何核心代码（尤其是 `search_bot.py`, `backup.py`, `sync.py`）前，必须遵循以下准则：
>
> 1. **修改前必关机**：严禁在 Bot 运行状态下直接覆盖关键代码。Agent 修改代码前 **必须先停止** Bot 进程。
> 2. **关机规范 (Shutdown)**：
>    - **Agent 操作**：必须先运行 `python src/utils/send_offline.py` 发送下线通知，然后再执行 `taskkill` 终止进程。
>    - **状态闭环**：确保管理员在 Telegram 私聊中收到“机器人正在下线”的通知，以防止产生未定义行为。
> 3. **变更后重启**：所有核心代码变更后，Agent 有义务重启 Bot，并协助核对新的 `RunID`。
>
> **禁止行为**：严禁跳过通知步骤直接 `taskkill`。

## 模式一：🔄 同步模式 (`/sync`)

**目标**：将订阅频道的新媒体资料完整地转发搬运到您的私有群组中。
**说明**：这是主要的工作流，遵循「本级来源」原则。**执行任一同步模式前，系统均要求显式的人工确认 (y/n)**。模式说明如下：

- **1. 局部更新同步**：增量同步，仅同步选定文件夹下的新资源。选择文件夹后进入频道列表逐一勾选（默认不勾选）。
- **2. 局部全时间轴**：从起始位置 #1 开始同步选定文件夹下的所有资源。同样使用文件夹列表选择。
- **3. 全局更新同步**：增量同步，针对配置文件中指定的所有全量频道。
- **4. 全局全时间轴**：从起始位置 #1 开始同步所有全量频道。
- **5. 高级回滚 (Rollback)**：根据同步历史记录，撤回目标群组的转发并清理本地元数据。
- **6. 同步情况一览**：展示各频道最近同步号和时间戳。页面内提供 **🔄 切换为🧪测试/🚀正式模式** 按钮。
- **🔄 切换为🧪测试/🚀正式模式（主菜单无编号按钮）**：控制同步任务是标记为 `TEST-X` 还是正式编号（如 `#20`）。
- ⚡ **处理速度说明**: 受限于 Telegram 官方的防刷屏速率限制 (Flood Wait)，同步速度大约为 **10~30 条消息/分钟**。如果涉及大量高分辨率视频转发，会自动触发保护机制导致耗时进一步增加。

> [!IMPORTANT]
> **同步限制**：同步模式会自动跳过机器人 (Bot)、私聊 (Private Chat)、好友 (Contact)、电报官方号 (Official)、收藏夹 (Saved Messages) 以及**已归档 (Archived)** 的项目。该模式仅针对具备广播性质的活跃“频道 (Channel)”和“群组 (Group)”进行资源同步。
>
> **⚠️ 全时间轴同步 (Full Time-axis Sync) 覆盖警告**：
> 当您对一个**已有历史同步记录**的频道重新发起“全时间轴同步”时，系统将以这次同步为唯一基准线。这会产生副作用：目标群组中可能出现大量重复转发消息，且本地旧日志虽保留，但其内部资源编号体系不再有效连续。
> **最佳实践**：如果需要重置频道同步状态，**强烈建议先使用高级回滚 (5)** 撤销至 `TEST-0`（或相应源点），这会自动清理历史残余，接着再执行新一轮的全量同步。若遇基准线冲突，控制台会弹出 `[高危操作警告]` 请求手动确认 (y)。
>
> **日志顺序**：同步日志（logs）存放在 `docs/archived/logs`，按**时间正序**排列；备份文档（backups）按**时间倒序**排列。
>
> ### 📊 消息分类与统计规则
>
> 为确保统计数据与编号逻辑自洽，系统将消息划分为两大核心类别：
>
> 1. **带资源消息**：指具备实际媒体载体（视频、图片、GIF、文件）或具备网页预览图（推特、YouTube 预览等）的消息。在汇总报告中体现为“带资源（组/预览）”。
> 2. **文本消息**：指纯文字消息，或包含网址但无网页预览图的链接消息。此类消息计入“文本消息数量”。
>
> **逻辑联动**：
>
> - **总消息数量** = 带资源消息 + 文本消息。
> - **携带链接消息**：只要消息内部含有 URL（无论其属于“资源”还是“文本”类别），均会独立计入“携带链接消息”并分配专属链接号。
>
> ### 🏷️ 编号图鉴 (Numbering Reference)
>
> **📍 编号展示位置差异**
>
> 系统会在以下三处主要区域打上标签与编号，为了平衡阅读体验，它们的展示粒度有所不同：
>
> 1. **Bot 发送的转发头和总结 (Telegram 内)**：仅包含消息容器级别的外层编号，保持界面清爽。
> 2. **同步日志 (docs/archived/logs)**：作为严谨存档，**除容器外，额外多出了对具体资源的细分独立编号**（例如 `🖼️ (2026-02-22 07:13) - vince 图片 #1`）。
>    - **显示顺序**：媒体组内容采用**倒序渲染**，上方优先聚合展示发布来源与长文本正文，在下方矩阵统一列出每项资源的对应编号标志与文件名。有效防止大段文本截断破坏附件流水号的连续性。
> 3. **备份日志 (docs/archived/backups)**：与归档 logs 保持严格一致最细粒度、一致顺畅显示排版。
>
> **🔢 标签字典**
>
> 消息头部采用多维编号系统，显示顺序如下：
>
> | 标签名称          |   图标   | 含义                              | 对应统计项     |
> | :---------------- | :------: | :-------------------------------- | :------------- |
> | **第 X 组消息**   |    📦    | 消息的物理索引（转发顺序）        | 总消息数量     |
> | **资源**          |    📦    | 携带媒体或预览的消息级编号 (互补) | 带资源消息数量 |
> | **文字**          |    ✍️    | 纯文本或独立链接的消息级号 (互补) | 文本消息数量   |
> | **带链接消息号**  |    📎    | 任何包含 URL 的消息               | 携带链接消息   |
> | **总资源号**      |    🔢    | 所有媒体文件的全局物理计数        | 资源总量明细   |
> | **视频/图片/GIF** | 🎬/🖼️/🎞️ | 特定媒体类型的分项编号            | 资源总量明细   |
> | **文件**          |    📄    | 文档、压缩包等非流媒体资源        | 资源总量明细   |
> | **可预览链接号**  |    👁‍🗨️    | 具备网页预览小窗口的链接          | 资源总量明细   |
> | **链接号**        |    🔗    | 全局 URL 独立计数                 | -              |
>
> **显示顺序**：第 X 组消息 > (资源/文字) > 带链接消息号 > 总资源号 > 分项明细。

### 📊 备份模式统计规范 (Backup Mode Stats)

为解决备份与同步模式间的统计对齐，系统在备份模式中采用以下双重指标：

1. **消息数量 (Message Count)**：最细粒度的统计口径，对应每一个独立的 Telegram 消息。这是**主要展示标准**。
2. **条目组数 (Item Group Count)**：将属于同一相册 (Media Group) 的多条消息聚合为 1 组后的数量。在 Markdown 文档中作为参考展示。

**命名规范**：

- **编号前缀**：生成的本地文件均带有 `#B` 开头的任务编号（如 `backup_#B12_...`）。
- **明细展示**：备份完成后，系统会推送明细报告，列出每个频道 `+本次新增 / 存档总计` 的消息条数。

### 手机控制方式

直接向 Bot 发送：

```text
/sync
```

### 命令行（服务器）手动控制

如果您无法使用手机端或者想在服务器上手动执行，可按照以下三步走：

1. **刷新元数据归档** (全量同步文件夹与状态)
   // turbo

```bash
python src/sync_mode/update_docs.py
```

2. **执行同步**

```bash
python src/sync_mode/sync.py
```

3. **再次刷新 (自动联动)**
   // turbo

```bash
python src/sync_mode/update_docs.py
```

> [!TIP]
> **智能移动特性**：`update_docs.py` 会自动检测频道文件夹变动并直接移动本地文件，无需手动清理旧目录。

## 模式二：💾 备份模式 (`/backup`)

**目标**：拉取源频道的历史消息记录（自动聚合相册），保存进本地的 SQLite 全局数据库 with 本地 MD/JSON。**不往目标群组发送/转发任何消息**。
**说明**：备份模式支持**局部树状图多选**和**全局全量增量**两种子模式。

- ⚡ **处理速度说明**: 极快。因为仅通过 Telegram API 拉取历史文本和数据库编号索引元数据，不需要等待服务器的媒体转发/重传响应。平均处理速度可轻而易举达到 3,000 ~ 5,000 条消息/分钟。
- **💡 数据主导逻辑**: 备份模式不仅是导出文档，更是**全量资源索引站**。
  - **任务编号制**: 每次备份分配唯一编号（如 `#B1`），文件名包含编号以方便追溯。
  - **精细化明细**: 备份总结会列出参与备份的所有群组及其各自的新增消息与总数。
  - **首见即登记**: 无论通过 `/backup` 还是 `/sync` 首次扫描到消息，系统都会立即为其分配永久 ID。
  - **归档一致性**: 备份 MD 文件中的“总消息数量”与数据库记录实时对齐。
  - **🔄 增量接力与自愈机制 (Incremental & Self-healing)**: **每一次备份生成的 JSON 都是该频道的“完整历史快照”**。当下一次执行增量备份时，系统遵循以下逻辑：
    1. **精确路径定位**：启动前自动扫描 `data/metadata` 建立 `ID -> 物理路径` 索引。即便频道改名导致文件夹变动，系统也能 100% 精确找回历史数据，绝不重爬。
    2. **自动剔除残缺**：系统会自动识别并忽略文件名以 `_PARTIAL` 结尾的文件，回溯搜索离当前时间最近的一个**完整备份点**作为断点。
    3. **接力合并**：提取断点 ID 后的新消息，与旧快照合并生成一份全新的全量快照。这确保了增量逻辑极其健壮，只需保证目录中有一个完整的 JSON 即可实现无限接力。

1. **局部备份**：强制同步选定频道的历史，覆盖元数据。
2. **局部增量更新备份**：仅追加选定频道自上次备份之后的新消息。
3. **全局备份**：针对所有已关注频道执行一次完整历史扫描。
4. **全局增量更新备份**：针对所有已关注频道执行一键增量追加任务。
5. **备份情况一览**：展示各频道最近备份号和时间戳。
6. **管理备份**：清理、瘦身或删除历史备份。

### 命令行（服务器）手动控制

```bash
python src/backup_mode/backup.py 窒物者
```

### 失联/封禁频道记录

如果你发现有某些已关注的频道已经被彻底封禁（或者被踢出），而你：

- **之前执行过备份/同步**：系统将自动对比 `data/metadata`，并在 `subscriptions.md` 底部的 `🚷 失联 / 封禁频道` 区块标识出来。
- **还未执行任何备份就被封**：你可以在 `data` 目录下手动创建并维护 `banned_channels.json` 文件：
  ```json
  {
    "-10012345678": {
      "name": "频道全称",
      "folder": "自定义分组名",
      "type": "Channel"
    }
  }
  ```
  下次执行 `update_docs.py` 时，该频道将被强制纳入失联名单供你归档查阅。

### 孤儿备份 (Orphan Backups) 自动/手动清理提示

当在 Bot 中执行 **“🗑️ 删除历史备份 (按批次)”** 时，系统会遍历该批次（如 `#B1`）下的所有文件并执行物理删除：

- 如果一个频道**还在你的关注列表中**（在线可用），它的备份文件会被正常追踪并被随批次删除。
- **特例：**如果一个频道已经被封禁或你已退群，它的历史备份文件会变成“孤儿”。此时：
  - 系统在全盘清理指定批次时，仍会尝试通过正则匹配前缀（如 `B1_`）把这些孤儿文件一同清理掉，**防止本地磁盘被失联频道的垃圾数据塞满**。
  - 但是，如果你发现 `data/metadata/` 或者 `docs/logs/` 下仍然残留有已封禁频道的无用数据，你可以放心**手动通过文件管理器删除**对应频道的 `.json` 和 `.md` 文件，不会影响数据库或其他频道的正常运行。

## 模式三：🔍 检索分析模式 (`/search`)

**目标**：对本地已归档、已备份的数据进行统一的全局检索查询。
**说明**：不进行抓取，只在本地数据库中查询。结果提供 **Deep Link (直达链接)**。

### 手机控制方式

发送 `/search` 进入菜单：

1. **更新数据库**：执行增量索引。
2. **找创作者**：交互式输入名字搜索。
3. **找模特**：交互式输入名字搜索。
4. **关键字搜索**：万能模糊匹配。

### 命令列控制

```bash
# 提取新候选词
python src/search_mode/update_search_db.py --extract
# 应用索引
python src/search_mode/update_search_db.py --apply
```

## 模式四：📥 手动补充信息 (`/ingest`)

**目标**：为已同步的消息手动“打标签”或“加注释”。
**说明**：将消息转发给机器人，然后使用追加功能。

- **互动打标**：支持多次追加文本。
- **检索融合**：追加后的内容通过 Mode 3 可搜。

### 手机控制方式

向 Bot 发信 `/menu` 选择 4 选项，转发消息后选择“追加/修改描述”。

## 📚 文档维护规则 (Documentation Maintenance Rules)

> [!IMPORTANT]
> **自动维护原则**：在每一轮与 AI 助手的对话开始及结束时，AI 必须自动评估本次代码变更、结构调整或逻辑优化是否影响了 `README.md` 或 `.agent/workflows/` 中的文档内容。
>
> **更新触发条件**：
>
> - 增加了新的 CLI 参数或手机 Bot 指令。
> - 调整了文件存储路径或数据库结构。
> - 修改了核心算法逻辑（如增量同步逻辑、资源分类规则）。
> - 引入了新的 UI 交互流程（如树状图、新菜单）。
>
> **操作要求**：一旦评估需要更新，AI 必须在任务清单中增加“文档同步更新”项，并在任务结束前完成对 `.md` 文件的修正，确保文档始终与最新代码保持同步。
