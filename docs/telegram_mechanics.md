# 📱 Telegram 官方底层机制说明

> [!NOTE]
> 这里的记录是关于 Telegram 平台官方的 API 行为。理解这些机制有助于看懂本项目的备份报告与原始计数逻辑。

## 1. 原始消息 (Raw Messages) 到底是什么？

在 Telegram 协议层，“原始消息”是 API 传给机器人的 **最小处理单元**。

### 核心规则：一条消息 = 一个资源

Telegram 的规则是：**一个 `msg_id` 只能携带一个媒体文件。**

- 如果你发一个视频 + 一段文字：这是一个 `msg_id`，算 1 条原始消息。
- 如果你发一个包含 10 个视频的相册：这在 API 层级实际上是 **10 条独立的消息**，每一项都有属于自己的 ID。

## 2. 相册/媒体组 (Albums / Media Groups)

虽然 API 层级是分开的，但 Telegram 通过 `grouped_id`（媒体组 ID）将它们关联在一起。

- **原始消息级别**：如果你一口气发送 10 张照片，API 会推送 10 个记录。这就是项目报告中 **“原始消息条数”** 的来源。
- **用户感知级别**：机器人根据 `grouped_id` 把这 10 条消息识别为一个整体帖子。在报告中，这被称为 **“1 组 (Group)”**。

## 3. 资源计数逻辑

在项目中，你会看到以下数值：

- **原始消息条数**：技术层面的“扫描件”总数。对于下载和索引而言，这是最精准的计数单位。
- **消息数量 (已合并)**：阅读层面的“帖子/动态”总数。
- **资源总量**：通常等于原始消息中携带媒体的数量。

## 4. 为什么进度条显示原始消息？

因为机器人下载时是按“原始条数”去服务器数数的。在扫描阶段，机器人无法预知后续还没抓到的消息哪些会属于同一个相册，因此 **“原始 ID 差值”** 是唯一可靠的进度预估标准。
